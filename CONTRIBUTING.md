# Contributing to yauzl

## Intro

Thanks for your interest in contributing to this project!
I'm a team of one person reviewing issues and PRs and publishing updates to this code.
I appreciate your patience, as I might take weeks-to-months to respond.

This project is "funded" strictly by my enthusiasm, and many aspects of its design are optimized for maintainer joy:
correctness, precision, test coverage, performance, thoroughly documented API, etc.
The best way that you, prospective contributor, can support the project is honestly just having a warm attitude.
We're all here making open source software to make the world a better place, and I appreciate you joining me on this journey! ❤️

## AI Policy

Using the model proposed in https://www.visidata.org/blog/2026/ai/ , PRs submitted to this project are limited to Levels 0-5,
meaning you need to fully understand everything you are presenting regardless of how it got written.

I am always operating at Level 0. I don't have any accounts with AI companies, because LLM-assisted code authoring takes the fun out of it for me.
I insist on understanding all the code I ship, and I ask that you respect my level of effort before asking me to review your code.

## When to open an Issue vs a PR

Simple typo fixes are pretty trivial to fully understand, but complex fixes or features are much harder.

If you don't fully understand the PR you're drafting, consider just filing a bug report or feature request instead.
I don't expect you to have full context on this project's history and nuances, so come on over to the issues section and let's discuss it!
I'd much prefer you ask for help than propose a change with misplaced confidence in an LLM.

Even if you do achieve full understanding of a complex PR, I may review your code and decide to rewrite it all in my own style anyway.
That doesn't mean you wasted your time! Showing interest in this project contributes directly to the "funding" mentioned above; I appreciate it!
However, if you don't like the idea of your code getting thrown away, let's have a design discussion in the issues before you invest your time in any code changes.

## Software engineering fundamentals

I won't articulate the fundamentals of how to open good bug reports, feature requests, or change requests in this guide.
These are skills that apply to any open source repo, so you should be able to find resources elsewhere (for example, bug reports should include steps to reproduce).
If you wish there were helpful links listed here and you've found some you like, feel free to open an issue or PR with suggestions!
